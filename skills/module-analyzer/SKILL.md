# C/C++ 模块深度分析技能

面向内部大型 C/C++ 项目的模块级分析工具。两种模式：根据目录路径生成模块设计文档，或根据函数名做影响面分析。

## 模式识别

根据用户输入自动判断模式：

**模式一：模块设计文档**
- 触发条件：用户提供目录路径（如 `src/network/http/`、`modules/auth`）
- 目标：生成完整的模块设计文档
- 交互：轻交互，扫描后问 1-2 个问题确认方向

**模式二：影响面分析**
- 触发条件：用户提供函数名（如 `HttpServer::handleRequest`、`parse_config`）
- 目标：追踪调用链、影响传播、给出改动建议
- 交互：确认影响追踪范围

如果输入同时包含目录和函数名，优先走模式一。

## When NOT to Use

- 分析整个项目（使用 repo-analyzer 代替）
- 单文件分析或简单的代码审查
- 调试具体问题（非架构理解类需求）

## 输出语言

默认中文。如果用户使用其他语言提问，则跟随用户语言。

## 输出位置

所有输出保存到 `~/repo-analyses/{module-name}-{YYYYMMDD}/` 目录。

## 前置条件

- 运行环境需配置 clangd LSP
- LSP 需能正常响应 `goToDefinition`、`findReferences`、`hover` 请求
- 如果 LSP 不可用，agent 应在启动时警告用户，并降级到 grep 模式（精度降低）

**灵活性原则**：以下所有阶段和章节都是建议性的指引，不是必须严格执行的清单。agent 应根据当前模块的特性动态决策——如果某个阶段对当前模块没有意义，可以跳过或简化。一切以最终文档的质量为准。

## 核心原则

### 1. 函数级精度

所有依赖关系和调用链必须精确到函数级别，标注源文件路径和行号。详细 LSP 使用方法参考 [dependency-analysis-guide.md](references/dependency-analysis-guide.md)。

### 2. LSP 优先，grep 兜底

- **第一步总是尝试 LSP**——它能理解编译上下文、宏展开、条件编译
- **LSP 失败时降级到 grep**——用正则匹配函数调用模式，结果需标注"未经验证"
- 两种来源的结论需标注来源：`[LSP]` 或 `[grep]`

### 3. 分层输出

核心函数展开完整分析，辅助函数只列目录。判断标准：
- **核心函数**：在主业务流程路径上、被多个外部模块调用、调用链深度 >= 3
- **辅助函数**：工具函数、getter/setter、简单的错误处理封装

### 4. 代码为准，标注溯源

- 每个结论必须标注源文件路径和行号：`src/http/parser.c:142-168`
- 所有函数调用关系必须通过 LSP 验证，不允许仅凭命名推断
- Mermaid 图中每个节点标注文件路径

### 5. 批判性思考与深度洞察

分析目标是让开发者**能上手工作、能做正确的修改**，而不是获得一份函数清单。

**评价和洞察**参考 [analysis-guide.md](references/analysis-guide.md)。每个设计决策要回答"为什么这样设计"和"不这样会怎样"，区分"设计选择"与"设计缺陷"。

**写作风格**：
- 有温度 — 像资深工程师给新同事做 onboarding，避免 AI 味套话
- 拒绝流水账 — 每个章节体现深度细节和设计洞察，不能泛泛而谈
- 行文流畅易懂 — 让不熟悉模块的开发者也能看懂并学到东西

## 模式一：模块设计文档工作流

### 阶段 M1: 目录扫描与结构识别

1. 验证用户提供的目录路径存在
2. 统计目录下的文件和代码规模：
   - 文件数量（.c/.cpp/.h/.hpp 分别统计）
   - 有效代码行数（排除空行和注释）
   - 子目录结构
3. 使用 `documentSymbol` 扫描所有源文件，提取函数/类型定义列表
4. 阅读模块自带文档和注释：
   - 模块内的 README、DESIGN、ARCHITECTURE 等文档文件
   - 头文件中的块注释（通常包含设计说明）
   - HACK/WORKAROUND/FIXME 注释（揭示已知设计约束）
   - 这些文档是理解"为什么这样设计"的第一手资料，比从代码推断更可靠
5. 识别模块边界信号：
   - 公共头文件（被模块外文件 include 的头文件）
   - 内部头文件（只被模块内 include 的头文件）
   - 入口函数候选（非 static、被模块外调用的函数）

### 阶段 M2: 轻交互确认

基于扫描结果，使用 AskUserQuestion 向用户确认（最多 2 个问题）：

1. **入口函数确认**：列出识别到的入口函数候选，让用户确认或补充
2. **分析重点确认**（如入口函数已很明确则可跳过）：确认是否有特定关注的子模块

### 阶段 M3: 函数级依赖梳理

**这是核心阶段，大量使用 LSP。** 详细方法参考 [dependency-analysis-guide.md](references/dependency-analysis-guide.md)。

#### 3.1 公开接口识别

1. 查找被模块外 `.c` 文件 `#include` 的头文件，提取其中声明的函数
2. 排除所有 `static` 函数
3. 用 `findReferences` 确认是否有模块外的调用者
4. 结果分为：`[公开接口]`、`[内部实现]`、`[待确认]`

#### 3.2 核心函数识别

1. 对每个非 static 函数用 `findReferences` 统计被调用次数
2. 对入口函数用 LSP 追踪调用链深度（递归追踪 3 层）
3. 标记规则：被外部调用 >= 2 次 → 核心；在入口函数调用链上 → 核心；调用链深度 >= 3 → 核心；其余 → 辅助

#### 3.3 依赖关系构建

对每个核心函数：
1. **正向依赖**（它调用了谁）：从函数体中定位所有函数调用点，用 `goToDefinition` 确认目标函数
2. **反向依赖**（谁调用了它）：用 `findReferences` 找所有调用点
3. **外部依赖边界**：调用到模块外的函数时，记录调用点和目标函数签名，用 `[跨模块]` 标注，不继续深入

对辅助函数：只记录函数签名和一句话职责（通过 `hover` 获取），不展开调用链。

### 阶段 M4: 端到端流程梳理

从用户确认的入口函数出发，追踪完整调用链：

1. 从入口函数开始，按调用顺序递归展开核心函数调用
2. 对每个分支标注：函数名 + 文件路径:行号、调用条件
3. 在模块边界处标注上下游：
   - `[上游: xxx_module::yyy()]` — 数据从哪来
   - `[下游: zzz_module::www()]` — 数据传给谁
4. 生成 Mermaid 时序图/流程图

### 阶段 M5: 并行深度分析（subagent 团队）

如果模块 > 50000 行，使用 Agent 工具并行启动 subagent 分析子模块。

#### 调度策略

- 按子目录或功能聚类划分子模块（每个子模块 1-3 万行为宜）
- 每个子模块 → 一个独立 Agent subagent（使用平台默认 agent 类型，不指定 subagent_type）
- 所有 subagent 在同一消息中并行启动

#### Subagent Prompt 要点

每个 subagent 必须获得：
- 模块整体设计哲学和架构概览
- 该子模块的文件范围
- 该子模块在整体中的位置（上游/下游模块）
- 公开接口列表（已从阶段 M3 获得）
- LSP 使用指令和覆盖率要求
- 叙事上下文（前一个子模块讲了什么、本模块需要铺垫什么）

#### 写入策略

- 每完成一个子系统/子模块的分析后，立即将该部分写入草稿文件
- 第一个子系统用 Write 创建文件，后续子系统用 Edit 追加
- 单次写入不超过 300 行
- 不要等全部文件读完再一次性写入

#### 主 Agent 等待纪律

- subagent 启动后，主 agent 不阅读 subagent 负责的源码
- 等待期间可准备报告骨架、处理外部依赖分析
- 所有 subagent 完成后才开始合并

### 阶段 M6: 质量验证

1. **覆盖率检查**：读取每个草稿末尾的覆盖率明细表，未达标的模块由主 agent 补充阅读
2. **LSP 结论抽查**：从每个子模块选取 2-3 个调用关系，回到源码验证准确性
3. **依赖图一致性**：检查跨子模块的调用关系是否一致（A 说调用了 B，B 的反向依赖里有没有 A）

### 阶段 M7: 文档生成

将所有分析结果融合为一份完整文档，写入 `$WORK_DIR/MODULE_DESIGN.md`。

不使用固定章节模板，但必须满足以下骨架约束：

**骨架约束**（文档必须覆盖以下内容，组织方式可灵活调整）：
- **模块概览**：在系统中的角色、目录结构、核心职责
- **核心数据结构**：关键 struct/enum/typedef、字段设计意图、数据结构间关系
- **函数索引**：区分公开接口、内部核心函数、辅助函数，标注文件位置、职责、调用者数量或调用链深度
- **函数级依赖图**：核心函数调用关系的 Mermaid 图，跨模块调用用虚线标注
- **外部依赖**：依赖的其他模块（调用方式和关键调用点）、依赖的系统库
- **被依赖方**：谁在使用本模块、调用了哪些函数
- **端到端流程**：从入口到出口的完整调用链，含错误处理路径和上下游边界标注
- **关键设计决策**：每个重大设计选择的动机、权衡和改进建议

分析思维和评价框架参考 [analysis-guide.md](references/analysis-guide.md)。

**叙事连贯**：文档不应是独立章节的拼接。根据模块特性选择叙事主线——数据流驱动（请求从进入到离开经过哪些函数）、分层驱动（从底层数据结构到上层业务逻辑）、或问题驱动（从模块要解决的核心问题到具体实现）。相邻章节之间要有过渡，而非"接下来看 X"式的生硬转折。例如："核心数据结构定义了连接的状态表示，但状态本身不会自己变化——驱动状态流转的是事件处理循环，下面展开其调用链。"

**写入策略**：文档可能较长，先 Write 前几个章节（200-300 行），后续用 Edit 追加，每次追加前 Read 确认末尾位置。

### 草稿文件

所有中间过程保存到 `$WORK_DIR/drafts/`：

| 阶段 | 文件 |
|------|------|
| M3 | `M3-dependencies.md`（依赖梳理结果） |
| M4 | `M4-flows.md`（端到端流程草稿） |
| M5 | `M5-submodule-{name}.md`（subagent 生成） |
| M6 | `M6-validation.md`（验证记录） |

## 模式二：影响面分析工作流

### 阶段 I1: 函数定位

1. 解析用户输入的函数名（支持 `Class::method`、`function_name`、`namespace::function` 格式）
2. 使用 `workspaceSymbol` 搜索函数定义位置
3. 如果找到多个定义（重载、不同文件），列出所有候选让用户选择
4. 使用 `hover` 获取完整函数签名（参数类型、返回值）

### 阶段 I2: 轻交互确认

使用 AskUserQuestion 确认分析范围（1 个问题）：

提供三个选项：
- A) 仅直接调用者（一层）
- B) 递归追踪 3 层
- C) 完整传播链（尽可能深）

### 阶段 I3: 调用链追踪

详细方法参考 [impact-analysis-guide.md](references/impact-analysis-guide.md)。

#### 正向追踪（它调用了谁）

1. 读取函数体，识别所有函数调用点
2. 对每个调用点用 `goToDefinition` 定位目标函数
3. 递归展开目标函数的调用（按用户选择的深度）
4. 记录每个节点的：函数签名、文件路径:行号、调用条件

#### 反向追踪（谁调用了这个函数）

1. 用 `findReferences` 找到所有调用点
2. 对每个调用者，获取其函数签名和定义位置
3. 递归追踪调用者的调用者（按用户选择的深度）

#### 影响传播树

综合正反向追踪结果，构建影响传播树。

### 阶段 I4: 影响分析与改动建议

1. **影响面评估**：影响的模块数量、调用路径数量、是否涉及关键路径
2. **改动风险分析**：修改函数签名 / 修改返回值语义 / 修改内部行为 / 添加新参数
3. **改动建议**：推荐的修改方式、需要同步修改的文件列表（带行号）、建议新增的测试点

### 阶段 I5: 输出

直接在对话中输出分析结果，或保存到 `$WORK_DIR/impact-{function-name}.md`。

## 通用规范

### LSP 使用规范

- **每次 LSP 调用必须检查返回值**——clangd 可能返回空结果（编译数据库未覆盖、条件编译分支）
- **LSP 超时处理**——单次 LSP 调用超过 5 秒视为超时，降级到 grep 并标注
- **结果去重**——`findReferences` 可能返回同一行的多个结果（宏展开），需要去重
- **跨文件追踪**——调用链跨越到模块外时，记录边界信息但不再深入

### 大模块处理策略

- 单文件 > 5000 行：分段读取（前 100 行类型定义 + import、核心函数按 `documentSymbol` 定位）
- 模块总代码 > 50000 行：启动 subagent 并行分析
- subagent 之间通过草稿文件通信，主 agent 负责合并

### 覆盖率要求

- **覆盖率** = 通过 Read 工具实际读取的行数 / 文件总行数
- **核心函数 >= 90%，辅助函数 >= 50%**
- 对于大文件（>500 行），必须读取：
  - 头部类型定义和导入（前 100 行）
  - 核心业务逻辑函数（通过 `documentSymbol` 定位）
  - 尾部测试代码（如有）
- 自动生成代码可降低覆盖率要求，扫描结构即可
- 每份草稿末尾必须附覆盖率明细表：

| 文件名 | 总行数 | 已读行数 | 覆盖率 | 未读原因 |
|--------|--------|---------|--------|---------|
| ... | ... | ... | ...% | ... |
| **合计** | ... | ... | ...% | ✅/❌ |

### Mermaid 图规范

- 依赖图用 `graph TD`（从上到下），节点格式：`函数名\n文件:行号`
- 端到端流程用 `sequenceDiagram` 或 `flowchart TD`
- 跨模块调用用虚线 `-.->` 并标注 `[跨模块] 模块名::函数名`
- 每张图不超过 30 个节点，超出时分拆为多张图

## 常见错误

- **只列函数不解释设计意图**：文档变成函数清单，读者无法理解"为什么这样设计"
- **缺少跨模块依赖边界分析**：只关注模块内部，忽略了与上下游的交互方式和耦合度
- **对所有函数一视同仁**：核心函数和辅助函数没有区分，重点不突出
- **忽略端到端流程中的错误处理路径**：只画 happy path，遗漏了异常分支和恢复逻辑
- **依赖图过大难以阅读**：超过 30 个节点的图信息密度过高，应按层次或子功能拆分

## 质量检查清单

最终文档发布前，逐项检查：

- [ ] 模块职责用一句话能说清楚
- [ ] 公开接口和内部实现的区分有代码依据
- [ ] 核心函数和辅助函数有明确的区分标准
- [ ] 每个依赖关系都通过 LSP 验证并标注了来源
- [ ] 端到端流程包含了错误处理路径，不只是 happy path
- [ ] 跨模块边界标注了上下游
- [ ] 设计评价有具体的❌/✅对比，不是泛泛的"设计合理"
- [ ] 所有结论标注了源文件:行号
- [ ] Mermaid 图节点不超过 30 个，可读性好
- [ ] 不熟悉模块的开发者只读文档就能理解核心设计决策
