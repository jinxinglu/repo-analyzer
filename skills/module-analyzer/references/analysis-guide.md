# 模块级分析思维框架

## 核心立场

分析的目标是让开发者**能上手工作、能做正确的修改**，而不是获得一份函数清单。好的模块分析像一位资深同事在做 code review——有判断、有推理、有建议，读完后开发者知道哪些地方可以放心改、哪些地方有坑。

每个评价都需要：代码依据、对比基准、推理过程。"这个模块设计得不错"不是评价，"模块对外只暴露了 3 个函数，内部实现细节全部通过 static 隐藏，调用者不需要理解内部状态机——这种封装让模块可以独立重构而不影响调用方"才是评价。

## 分析技法

### 对比法

"如果是我来设计，我会怎么做？"——如果你的第一反应方案和项目的方案不同，深入比较两者的权衡。

> ❌ 该模块使用回调函数进行异步通知。
>
> ✅ 模块选择了回调函数而非轮询队列来通知事件完成。轮询队列实现更简单、调试更容易（事件在队列里可以看到），但会引入延迟（取决于轮询间隔）。回调的代价是调用时机不可控、回调中如果出错会导致调用方崩溃，但换来了零延迟的事件通知。对于这个实时性要求高的场景，回调是合理的选择。

### 追问法

"这个设计解决了什么不明显的问题？"——表面上看起来过度设计的地方，可能是在防御一个你没注意到的边界情况。

> ❌ 模块为每个连接维护了一个独立的状态机。
>
> ✅ 每个连接一个状态机看起来内存开销大——用一个全局状态表加连接 ID 索引更省内存。但追问后发现：连接的状态转换逻辑依赖连接的上下文（协议版本、协商的加密套件），拆成全局表意味着每次状态转换都要查上下文，反而更复杂。独立状态机把上下文和状态绑在一起，虽然内存多用了一点，但换来了每个连接的独立性——一个连接的状态出问题不会影响其他连接。

### 场景法

"在极端场景下会怎样？"——高并发、内存不足、网络中断、下游服务超时。好的设计在极端场景下优雅降级。

> ❌ 模块使用了固定大小的连接池（最大 1024 个连接）。
>
> ✅ 连接池上限 1024 看似武断，但用场景法验证：超过 1024 并发连接时，每个连接的内存开销约 4KB，总计约 4MB——这是一个可控的内存上限。如果改成动态增长，在高并发场景下内存消耗不可预测，可能导致 OOM。固定上限的代价是超过 1024 的新连接需要等待，但这比整个进程崩溃要好得多。

### 演进法

"这个设计是如何演变到现在的？"——git 历史和代码中的注释有时能揭示设计演进的故事，当前的"复杂"可能是多次踩坑后的智慧结晶。

当 git blame 显示某段代码经历了多次修改，或者注释中提到"HACK"、"WORKAROUND"、"FIXME"时，值得追踪这段代码的演进历史——这些标记往往是理解设计约束的钥匙。

## 模块职责评估

### 如何判断职责边界是否清晰

一个设计良好的模块有**一句话能说清楚的职责**。如果需要用"和"来描述，职责可能过多。

**信号检测**：
- 模块内存在两组函数，彼此之间没有调用关系 → 可能是两个模块被错误地放在一起
- 公开头文件里有不相关的函数声明 → 接口泄漏，模块边界模糊
- 修改某个功能时，总需要同时改另一个模块 → 职责划分可能有误

**评估方法**：
- 看公开接口：如果接口函数可以分成几个完全不交互的组，模块可能承担了过多职责
- 看内部依赖：如果模块内部存在明显的"子集群"（一组函数互相调用但很少和另一组交互），这是子模块拆分的信号
- 看命名一致性：好的模块，其函数名通常有统一的前缀或命名模式。命名混乱往往暗示职责混乱

> ❌ 该模块负责 HTTP 请求的处理。
>
> ✅ 该模块只负责 HTTP 请求的解析和序列化——它把字节流变成结构化的请求对象，把响应对象变成字节流。它不负责路由、认证、或业务逻辑处理。这个边界划分让解析模块可以独立测试（给字节流，检查输出结构），也可以在 HTTP 服务器和 HTTP 客户端之间复用。

### 职责边界问题的层次

| 层次 | 示例 | 严重程度 |
|------|------|---------|
| 微小的职责溢出 | 一个工具函数放在了业务模块里 | 低——可以容忍 |
| 明显的职责混合 | 日志功能和网络功能在同一模块 | 中——应该拆分 |
| 核心职责不清 | 模块既做数据解析又做业务决策 | 高——重构优先级高 |

## 耦合度评估

### 如何判断依赖是否合理

依赖本身不是问题，**不必要的依赖**才是。

**合理的依赖**：
- 只依赖稳定的接口（头文件中的函数声明），不依赖实现细节（内部数据结构）
- 依赖方向是单向的，不存在循环依赖
- 依赖的是抽象层而非具体实现

**危险信号**：
- 直接访问另一个模块的内部全局变量 → 最严重的耦合，修改一处到处爆炸
- 包含另一个模块的内部头文件（非 public 头文件）→ 依赖了实现细节
- 循环依赖：A 调 B，B 也调 A → 通常意味着职责划分有问题
- 依赖数量过多：一个模块依赖了超过 5 个其他模块 → 可能职责不清或缺少中间抽象层

> ❌ 该模块依赖了 3 个外部模块。
>
> ✅ 该模块只依赖了 logger 模块（通过 `logger_log()` 接口）和 config 模块（通过 `config_get_int()` 接口）。两个依赖都是通过稳定的函数接口调用，不直接访问对方的数据结构。但注意到 parser.c:245 直接 include 了 `event_internal.h`——这是 event 模块的内部头文件，意味着 parser 依赖了 event 的实现细节。如果 event 的内部数据结构改变，parser 也需要改。

### 耦合评估维度

对每个外部依赖，评估：
1. **依赖方向**：单向 ✅ / 双向 ⚠️ / 循环 ❌
2. **依赖粒度**：依赖接口 ✅ / 依赖数据结构 ⚠️ / 依赖全局状态 ❌
3. **依赖稳定性**：依赖稳定模块 ✅ / 依赖频繁变动的模块 ⚠️
4. **依赖必要性**：有明确业务需要 ✅ / 只是图方便 ⚠️ / 完全多余 ❌

## 数据结构评价

### 如何评价核心数据结构的设计

数据结构是模块的骨架。评价数据结构不是罗列字段，而是分析**为什么这样组织数据**。

**关注点**：
- **粒度**：struct 是太大（上帝对象）还是太小（碎片化）？好的粒度让每个 struct 有一个清晰的生命周期管理者
- **所有权**：谁负责分配和释放？指针指向的数据谁拥有？所有权不清是 C 项目中最常见的 bug 来源
- **不变量**：数据结构有哪些隐含约束（如"这个字段非空时那个字段必须为 0"）？不变量越少、越显式，代码越不容易出错
- **演化能力**：如果要加一个字段，需要改多少地方？好的数据结构设计让扩展变得局部化

**分析方法**：
- 追踪数据结构的生命周期：从创建、使用到销毁，沿途经过了哪些函数
- 检查是否有"万能 struct"——字段很多但不是每个场景都用所有字段，说明可能需要拆分
- 检查回调函数指针字段——它们暗示了策略模式的使用，值得展开说明

> ❌ 该模块使用 Connection 结构体表示连接。
>
> ✅ Connection 结构体（conn.h:15-42）是模块的核心数据结构，包含 23 个字段。其中 fd、state、last_active 是连接的生命周期管理字段，read_buf、write_buf 是 IO 缓冲区，callback 字段是注册的事件回调。值得注意的是，这个结构体的字段可以分为两组——一组由 event_loop 管理（fd、state、callbacks），另一组由业务逻辑管理（request、response）。这种混合意味着修改 event_loop 的实现可能影响业务逻辑，如果连接数量增长到需要多种 event_loop 实现（如 epoll/kqueue/IOCP），可能需要将 event 相关字段抽到独立的 EventConn 结构中。

## 流程分析

### 如何分析函数调用链

不是画完调用链就完了，要回答：**这条链路的设计合理吗？**

**关注点**：
- **调用深度**：从入口到最深处超过 7 层 → 可能需要中间抽象层
- **错误传播路径**：错误是怎么从底层传到上层的？是返回值逐层传播（显式但冗长）、是错误码全局变量（隐式但危险）、还是回调通知？
- **数据转换节点**：调用链中哪些函数在做数据格式转换？转换节点越多，性能开销越大，也越容易出 bug
- **同步 vs 异步边界**：调用链中哪里从同步变成了异步？这种边界往往是复杂性最高的地方

> ❌ 请求处理流程为：accept → read → parse → process → send → close。
>
> ✅ 主路径是 accept → read → parse → process → send → close（6 层，在合理范围内）。但错误传播路径值得关注：parse 阶段的错误通过返回 NULL 直接跳到 send_error（快速失败，好），process 阶段的错误通过设置 errno 继续执行（调用方需要检查 errno，容易遗漏）。这两种错误处理方式的不一致可能是历史原因——parse 是后写的模块，process 是早期代码。如果要统一错误处理，建议全部采用返回值方式。

### 端到端流程的质量评估

一条好的端到端流程应该：
- 主路径清晰，读者能一眼看出正常情况下的数据流向
- 错误处理路径不掩盖主路径（不是每个函数调用后都跟 10 行错误处理）
- 分支逻辑有清晰的层次（先判断大方向，再处理细节），不是"意大利面条"式的跳转

## 设计缺陷识别

### 如何区分"设计选择"和"设计缺陷"

**设计选择**：在特定约束下的合理权衡。可能有更好的方案，但当前方案有它的道理。
**设计缺陷**：导致实际问题的架构问题——bug 温床、修改困难、性能瓶颈。

判断方法：
1. **影响面分析**：这个问题影响了多少功能？影响面越大，越可能是真正的缺陷
2. **修改频率**：这个区域是否经常需要修改？经常改的地方如果设计不好，每次改都是痛苦
3. **bug 聚集**：这个区域是否是 bug 高发区？bug 聚集是设计缺陷的强信号
4. **演化风险**：如果项目规模增长 10 倍，这个设计还能撑住吗？

### 常见 C 项目设计缺陷模式

| 模式 | 信号 | 影响 |
|------|------|------|
| 上帝函数 | 单个函数 > 500 行，做了太多事 | 难以理解、测试、修改 |
| 全局状态泛滥 | 大量 extern 变量 | 并发不安全、修改影响不可预测 |
| 回调地狱 | 多层嵌套回调、回调中注册回调 | 控制流难以追踪 |
| 资源管理不统一 | 有些路径 free 了有些没有 | 内存泄漏、双重释放 |
| 错误处理不一致 | 有的函数返回错误码有的直接 abort | 调用方不知道该如何处理错误 |
| 紧耦合的初始化顺序 | 全局变量之间的初始化有顺序依赖 | 启动顺序错误导致崩溃 |

## 批判性思考指引

### 如何写出有价值的评价

- **有代码依据**：每个评价指向具体的文件和行号
- **有对比基准**：说"设计合理"时，对比的是什么？C 标准库的做法？同类项目？这个模块自己的设计目标？
- **区分"不同"和"不好"**：C 项目中非主流的设计选择（如自己实现链表而非用 glib）可能是性能考虑，不一定是缺陷
- **承认不确定性**：如果看代码推断出的结论可能有其他解释，标注出来——"从代码来看可能是 X，但也许有我没看到的约束"

### 评价的诚实性

- 模块设计好就多写亮点，不需要凑问题
- 有明显缺陷就直说，不需要找补
- 如果模块很小、很直接，如实说"这个模块实现简洁，没有复杂的架构决策"——不是每个模块都需要深度分析

## 写作要求

### 文风

- **有温度**：像资深工程师给新同事做 onboarding，加入主观评价和建议。避免"该模块实现了XXX功能"式的 AI 味套话
- **拒绝流水账**：每个章节体现深度细节和设计洞察，不能一句带过或泛泛而谈
- **行文流畅易懂**：让不熟悉这个模块的开发者也能看懂并学到东西，避免堆砌术语

### 深度标准

> ❌ 模块使用哈希表来存储连接映射，支持 O(1) 的查找操作。
>
> ✅ 连接映射选择了哈希表而非红黑树。红黑树是 O(log n)，但支持有序遍历——如果需要"找出所有超时连接"这样的范围查询，红黑树更合适。当前模块只需要按 ID 精确查找，不需要范围查询，哈希表的 O(1) 是正确选择。但哈希表的冲突处理用了链地址法而非开放寻址——在连接数接近桶大小时，链地址法的 cache 友好性差，如果连接量从 1000 增长到 100000，可能需要改为开放寻址或换用更高性能的哈希函数。

特征：有具体的技术推理、有量化的权衡、有"当前合理但未来可能需要调整"的前瞻性判断。
